# 识谱准确率提升计划 v2（高清 PDF + 音频双证据链）

> 2026-07-09 制定。背景：week09 试点 sys01 32/32 全对，但 sys02-05 仅 27%~68%，
> 卡点 = 密集段弦位读不准 + 模板库不全（playbook 已确认这是最大瓶颈）。
> 新增素材：真 300 DPI 扫描 PDF（旧版只有 ~150 DPI）+ 全书 105 条示范音频。

## 核心思路

准确率不靠单点优化，靠**三条独立证据链交叉对账**：

1. **图像链**（现有 NCC）：高清源重建后精度天花板直接抬高
2. **音频链**（basic-pitch）：弦错=纯四度音高偏移、品错=半音偏移，音频全都能抓到
3. **乐理链**（现有 tabcheck）：弦位可行性校验

三链一致 → 自动通过；任一失配 → 进人工复核清单。目标是把人工复核量压到每段 <3 处。

**sys04/sys05 关键发现（须 Phase 1 音频裁决，勿静默改）**：
经三重独立核验——① 读谱器的 6 线栅格与独立检测的印刷谱线逐条吻合（栅格无错位）；

结果不变——**读谱器读到的弦位与印刷 TAB 一致，但与现有 GT JSON 冲突**（sys05 下声部

GT 转写错误/歧义，而非读谱器错**。读谱器很可能比现 GT 更准。须用音频（Phase 1）或按高清图
重转 GT 来裁决，本阶段不动 GT、不硬调读谱器。

**fret 0**：sys04 GT 每小节 1 个"弦3品0"，但该位置印刷字形实为"9/10"，锚点全落在 9 上；
判定为可能的 GT 错。未入模板库 → 这些音诚实标"漏检-无模板(品0)"待复核，不臆造"0"模板。

**Phase 1.4 demucs 分离已建并试 → 结论：音频仍无法裁决 sys05（三源皆平局）**。
建成可复用管线：`work/separate_stem.py`（demucs htdemucs 底层 API + soundfile 存储，绕开
torchaudio/torchcodec 存储坑；装 demucs 4.0.1 到 Python311）→ `work/bp_on_stem.py`（干净声部跑
basic-pitch）→ `work/adjudicate_sys05.py`（逐槽排他投票）。**结果：贝斯已成功去除，但排他投票
三源全平**——全混 13:13、纯吉他(other) 14:13、去贝斯(no_bass) 13:12。根因＝sys05 是连奏段

叠加纯四度关系 + 吉他泛音重叠，音频无法区分弦5/弦6。**须由用户按印刷图/乐理裁决**：
印刷侧证据（两声部数字行距 48px≈2线距＝弦4+弦6）倾向弦6，但与标题"跨越⑤弦④弦"冲突。

用缓存 basic-pitch 事件对 sys05 建两假设栅格（GT弦5=E/F/G vs 读谱器弦6=B/C/D，差纯四度）做

## 调研来源

- 图像→tab 开源项目基本只有玩具级（[OCR-tabber](https://github.com/sn0v/OCR-tabber)），确认自研 NCC 路线是对的，Soundslice 仍是唯一对口商业方案
- 音频→tab：[TabCNN](https://github.com/andywiggins/tab-cnn)（CQT+CNN 直接出弦位）、[CRNN 复现（GuitarSet MPE F1=0.87）](https://github.com/trimplexx/music-transcription)、TART/GOAT 2025-2026 新论文（技巧感知转谱、大规模配对数据集）
- 音频→MIDI：[basic-pitch](https://github.com/spotify/basic-pitch)（已在用，复音+吉他表现力设计，单乐器最佳——伴奏干扰时配 demucs）
- 对账方法论：audio-to-score alignment 领域标准做法就是 DTW + onset 容差（50/100/200ms 三档），v4 节奏栅格思路与学术主流一致
